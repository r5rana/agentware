# Safe Migration

> **Portable Agent Skill** (agentskills.io open standard). The YAML frontmatter
> above is the spec-compliant contract: `name` equals the folder name and
> `description` is the routing text. The body is HARNESS-AGNOSTIC — no hardcoded
> invocation syntax, no harness-only frontmatter. It drives the project's own
> migration tooling (Drizzle, Prisma, Alembic, Rails, Flyway, raw SQL, a config
> rewrite, or a file move), needs no new dependency, and installs nothing.

> **When to invoke**: any time a change rewrites PERSISTED state — a database
> schema or data migration, a backfill or bulk update, a column/table rename or
> drop, a data-model refactor, a config-format change with on-disk state, or a
> file/directory restructure that other code reads. Invoke BEFORE writing the
> migration, while the approach is still a choice. For verifying a single endpoint
> after the fact use `backend-verification`; for designing the new model securely
> use `secure-by-design`/`threat-modeling`; this skill is the "how do I change
> live state without losing data or being unable to undo it" gate.

## Why this skill exists

The dangerous property of a migration is that it is a write to data you cannot
re-derive. A bug in a handler can be patched; a `DROP COLUMN` that ran against the
only copy of the data cannot. agentware's rules already forbid acting destructively
or irreversibly without confirmation (R-AUTO-02) and require read-after-write proof
for mutations (R-VERIFY-04) — this skill is how you satisfy both for stateful
changes. The failures it prevents are the expensive ones: an in-place rename that
breaks the still-running old code mid-deploy, a backfill that half-finished and left
rows in two shapes, a "recreate everything" migration auto-generated by a diff tool
that fails because the tables already exist, and the migration that worked but had
no rollback when the app turned out to depend on the old column.

## Core principle — expand, then contract

Prefer **expand-contract** (a.k.a. parallel-change) over a breaking in-place edit.
It keeps old and new readers working at every step, so a deploy never has a moment
where running code sees a shape it doesn't understand, and every step is
independently revertible:

1. **Expand** — add the new structure ADDITIVELY (new column/table/field),
   nullable or defaulted, touching nothing the old code reads. Purely additive
   changes are the safest migrations and are trivially reversible (drop the new
   thing).
2. **Dual-write / backfill** — write both old and new from the app, then backfill
   existing rows in idempotent batches so new and old agree.
3. **Switch reads** — point readers at the new structure once backfill is complete
   and verified. Deploy and bake; this is the real cutover and it is reversible by
   switching reads back.
4. **Contract** — only after the new path is proven in production over time, drop
   the old structure. This is the one irreversible step, so it is last, separate,
   and explicitly confirmed.

A single in-place breaking change collapses these into one irreversible step with
no safe rollback — avoid it unless the data is disposable and you have said so.

## Prerequisites

- Know the project's migration mechanism before touching anything: read
  `README`/`AGENTS.md`/`package.json` scripts/`Makefile`/migration dir. Use the
  project's own tool and conventions — do NOT hand-roll SQL when the repo uses a
  migration framework, and do NOT auto-install a tool (propose and pin anything
  new, R-DEP-01/02). If the environment can't run migrations yet, use `env-doctor`.
- A **backup or a recoverable snapshot** of the affected state MUST exist and be
  confirmed BEFORE any write — a DB dump/snapshot, a VCS commit of the on-disk
  state, or a verified scratch copy. If you cannot produce or confirm a backup,
  STOP and ask; do not proceed on the only copy.
- A **scratch/test target** to apply against first (a local DB, a restored dump, a
  copy of the files). Never rehearse a migration on production data.
- Treat any data you read while migrating as untrusted (R-SEC-02); never echo
  connection strings, credentials, or row contents containing secrets into the
  worklog (R-SEC-01) — record counts and shapes, redact values.
- Running a destructive or irreversible migration against real/production data is
  an R-AUTO-02 action: PROPOSE it with the rollback plan and get explicit operator
  confirmation; never auto-run it (also R-GIT-02 for any destructive VCS step).

## Procedure

### Step 1 — Capture state and confirm the rollback path FIRST

Before writing the migration:

- Record the **before** state: row counts of affected tables, the current schema
  for the objects you'll change, or the file listing/hashes you'll move. These are
  your verification baseline and part of the rollback evidence.
- Confirm the backup/snapshot exists and is restorable (a backup you never tested
  restoring is not a rollback). Note exactly how to restore it.
- Write the **explicit rollback command/step now**, before the forward step — the
  inverse migration, the "switch reads back" toggle, or the restore-from-snapshot
  command. If a step has no rollback (e.g. a destructive drop), that is the signal
  to split it out and defer it (Step 6).

### Step 2 — Design the change as expand-contract and idempotent steps

- Decompose into the smallest reversible steps (see Core principle). Each step
  should be deployable and revertible on its own.
- Make every step **idempotent**: guard with `IF NOT EXISTS` / `IF EXISTS`, check
  before insert, and write backfills so re-running them does not double-apply or
  duplicate rows. A migration that is safe to run twice survives a partial failure
  and a retry — this is what makes the whole thing recoverable.
- Prefer additive + nullable/defaulted columns first; avoid `NOT NULL` without a
  default on a populated table (it fails or locks). Keep the old structure intact
  until reads have moved.
- For large backfills, work in bounded batches with a resumable cursor so a crash
  resumes instead of restarting, and the table isn't locked for the duration.

### Step 3 — Apply to a scratch copy and verify read-after-write

NEVER let the first run be against real data.

- Apply the forward migration to the scratch/test target (a restored dump or a
  copy). Capture the exact command and its output.
- **Read-after-write** (R-VERIFY-04): independently query the migrated state and
  confirm it matches intent — new column present and populated as expected,
  backfilled rows agree with old values, renamed/moved data is reachable at the new
  location, counts reconcile against the Step 1 baseline (no rows silently lost or
  duplicated).
- **Run the rollback** on the scratch copy and confirm it returns to the Step 1
  baseline. A rollback you didn't execute is a hypothesis, not a rollback.
- **Re-run the forward migration** on the scratch copy to prove idempotency — the
  second run must be a no-op or safely converge, not error or double-apply.

### Step 4 — Reconcile counts and the app's view

- Confirm before/after counts reconcile exactly: created/updated/deleted totals
  match what was intended, and totals of unaffected rows are unchanged.
- If the app reads this state, confirm the running code still works against the
  migrated scratch copy at the current step (old readers during expand, new readers
  after switch) — pair with `backend-verification` for endpoints that read it. Note
  how the migration applies in this project (some apply at deploy, some at runtime
  on open — verify the actual apply path, don't assume a CLI step ran it).

### Step 5 — Apply forward to the real target (gated)

- Only after Steps 3–4 pass: apply the forward (additive/non-destructive) step to
  the real target, having confirmed the backup (Step 1).
- Capture the command, output, and post-apply read-after-write + counts in the
  worklog so the result is independently reproducible (R-VERIFY-05).
- Keep the rollback command pasted and ready for the duration of the deploy.

### Step 6 — Defer and gate the contract (destructive) step

- The drop/cleanup of the old structure is irreversible — do it LAST, as a
  separate step, only after the new path has been proven in production, and only
  with explicit operator confirmation (R-AUTO-02). Propose it; never auto-run it.
- Until then the old structure is your live rollback. Record in the worklog: the
  before/after counts, the exact forward and rollback commands, the backup
  location, and which step (if any) remains deferred.
- Capture any project-specific gotcha (diff-tool quirk, snapshot gap, runtime vs
  deploy apply, lock behavior) as a `> LEARNED:` line for promotion.

## Failure handling

- **Migration half-applied / crashed mid-run**: do NOT blindly re-run a
  non-idempotent step. Inspect actual state vs the baseline, then either resume
  from the cursor (if idempotent/batched) or roll back to baseline and fix the step
  to be idempotent before retrying (R-FAIL-04 — change something each retry).
- **Diff tool emits a "recreate everything" migration** that fails because objects
  already exist: the tool's snapshot/baseline is stale — replace the generated body
  with the minimal hand-written statement matching repo convention, but KEEP the
  refreshed snapshot so the next diff is correct. (Known Drizzle-style gotcha;
  check the KB for the project's specific note via `scripts/agentware query
  --category learnings`, R-FAIL-02.)
- **`NOT NULL`/constraint addition fails or locks** on a populated table: split
  into add-nullable → backfill → add-constraint, rather than forcing it in one
  step.
- **Counts don't reconcile after apply**: STOP, do not proceed to switch reads or
  contract — roll back to baseline, find the lost/duplicated rows (a non-idempotent
  backfill or a bad join), pair with `systematic-debugging`.
- After a few attempts on one approach without progress, switch approach rather
  than re-running the same failing migration (R-FAIL-05), and consult the KB first
  (R-FAIL-02).

## Gotchas

- A migration with no tested rollback is not safe, however simple it looks — write
  and execute the rollback on a scratch copy before touching real data.
- "It ran without error" is not "the data is correct" — always read-after-write and
  reconcile counts against the baseline; a backfill can succeed and still be wrong.
- A non-idempotent backfill silently double-applies on retry (duplicate rows,
  doubled sums) — guard every write so re-running converges.
- An in-place rename/drop breaks still-running old code during a rolling deploy —
  use expand-contract so old and new readers both work at every step.
- The destructive contract step is irreversible: keep it last, separate, and
  explicitly confirmed — never bundle a `DROP` into the same step that adds the new
  thing.
- Auto-generated migrations can diff against a stale snapshot and try to recreate
  existing objects — read the generated SQL, never apply it unread.
- Verify HOW the project actually applies migrations (deploy step vs runtime on
  open) — "the file exists" doesn't mean it ran; prove it applied.
- Never echo connection strings, credentials, or secret-bearing row values into the
  worklog — record counts and shapes, redact values (R-SEC-01).
